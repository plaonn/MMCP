# 이메일 벌크 작업 장애 복구

## 상태

대기 중

## 문제

복수형 이메일 변경 도구 실행 중 서버 장애, 네트워크 단절 또는 응답 유실이
발생하면 어떤 작업까지 성공했는지 확인하기 어려움. 특히 복사 작업을 그대로
재시도하면 중복 메일이 생길 수 있음.

## 원하는 동작

벌크 호출과 개별 작업 상태를 영속적으로 기록하고, 장애 후 확정된 상태와
불확실한 상태를 조회하여 안전한 작업만 재개할 수 있어야 함.

## 범위

### 포함

- 벌크 호출과 개별 작업의 영속적 실행 저널
- `pending`, `running`, `succeeded`, `failed`, `uncertain` 상태 관리
- 동일 벌크 ID와 작업 ID의 중복 실행 방지
- 벌크 상태 조회와 안전한 작업 재개 도구
- 읽음·별표 작업의 현재 상태 확인을 통한 자동 복구
- 이동·휴지통·스팸·복사 작업의 불확실 상태 처리

### 제외

- IMAP이 지원하지 않는 transaction 또는 정확히 한 번 실행 보장
- 불확실한 복사 작업의 자동 재시도

## 완료 조건

- [ ] 서버 재시작 후 미완료 벌크 작업 상태를 조회할 수 있음.
- [ ] 안전한 `pending` 작업만 명시적으로 재개할 수 있음.
- [ ] 결과를 확정할 수 없는 작업은 `uncertain`으로 남음.
- [ ] 복사 작업을 자동 재시도하여 중복 메일을 만들지 않음.

## 결정

- SQLite 기반 작업 저널을 우선 검토함.
- IMAP 성공 후 저널 기록 전 장애가 발생한 작업은 성공으로 추정하지 않음.

## 미결 사항

- 벌크 ID를 호출자가 제공할지 서버가 발급할지
- 실행 저널 보존 기간과 정리 정책
- 상태 조회 및 재개 MCP 도구의 정확한 계약

## 구현 참고

- 이메일 본문과 인증정보를 저널에 저장하지 않음.
- 복구 판단에 필요한 최소 식별자와 작업 인자만 저장함.
