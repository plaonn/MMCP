# 저장소 작업 지침

## 진실 공급원

- `docs/SPEC.md`는 시스템이 현재 제공해야 하는 동작을 설명함.
- `docs/ROADMAP.md`는 아직 확정되지 않은 장기 계획과 개발 순서를 설명함.
- `work/*.md`는 제안되었거나 진행 중인 변경 사항을 설명함.
- `docs/decisions/`는 장기간 보존할 설계 결정과 그 이유를 설명함.
- 소스 코드와 테스트는 `docs/SPEC.md`를 구현하고 검증함.
- Git 이력은 완료된 작업의 보관소임. 완료된 작업 파일을 유지하지 않음.

## 필수 작업 절차

동작을 변경하기 전:

1. `docs/SPEC.md`를 읽음.
2. 장기 계획이나 개발 순서와 관련된 작업이면 `docs/ROADMAP.md`를 읽음.
3. 관련된 `work/` 파일을 모두 읽음.
4. `work/TEMPLATE.md`를 기반으로 작업 파일을 생성하거나 갱신함.
5. 구현 전에 합의된 요구사항, 결정 사항, 완료 조건을 작업 파일에 기록함.

구현 중:

- 요구사항이나 결정이 바뀌면 작업 파일을 최신 상태로 유지함.
- 모든 동작 변경에 테스트를 추가하거나 갱신함.
- 요구 동작이 바뀌면 같은 변경에서 `docs/SPEC.md`도 갱신함.
- 장기 계획이나 개발 순서가 바뀌면 `docs/ROADMAP.md`도 갱신함.
- 정확한 기계 판독용 API 및 도구 스키마는 코드에 정의함. 스키마를 문서에
  수동으로 중복 작성하지 않고 필요할 때 코드에서 참조 문서를 생성함.
- `docs/SPEC.md`만으로 이유를 알기 어려운 장기 설계 결정은
  `docs/decisions/`에 기록함.

작업을 완료하기 전:

1. 모든 완료 조건을 검증함.
2. 관련 테스트를 실행하고 검증 결과를 작업 파일에 기록함.
3. 코드, 테스트, `docs/SPEC.md`가 서로 일치하는지 확인함.
4. 필요한 장기 결정 내용을 `docs/decisions/`로 옮김.
5. 완료된 작업 파일을 삭제함. Git 이력으로 보존됨.

## 일관성 규칙

- 프로젝트 문서와 작업 파일은 한국어로 작성함. 코드 식별자, 프로토콜
  고유명, 오류 문자열처럼 정확한 원문이 필요한 표현은 유지함.
- 코드와 `docs/SPEC.md`가 충돌하는 상태에서 작업을 완료하지 않음.
- 코드 동작이 불명확하면 테스트와 `docs/SPEC.md`에서 명시적으로 정의함.
- 이미 명시된 동작을 복구하는 버그 수정은 스펙 변경이 필요 없음. 요구
  동작 자체를 바꾸는 버그 수정은 스펙도 변경해야 함.
- `docs/SPEC.md`에는 현재 동작만 기록함. 구현 이력, 논의 기록, 추측성
  아이디어를 넣지 않음.
- `docs/ROADMAP.md`에는 아직 확정되지 않은 장기 계획과 개발 순서만 기록함.
  곧 논의하거나 실행할 구체적인 항목은 `work/`에서 관리함.
- 비밀값, 인증정보, 토큰, 실제 개인 이메일 데이터를 소스, 테스트,
  시험 데이터, 로그, 작업 파일, 문서에 넣지 않음.
