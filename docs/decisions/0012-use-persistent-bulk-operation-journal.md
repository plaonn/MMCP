# 0012. 영속 벌크 작업 저널 사용

## 상태

승인됨

## 배경

이메일 벌크 변경 중 서버 장애나 응답 유실이 발생하면 성공한 작업과 시작하지
않은 작업을 구분하기 어려움. 특히 복사 작업을 그대로 재시도하면 중복 메일이
생길 수 있음.

## 결정

- 6개 이메일 벌크 변경 도구는 호출자가 제공하는 UUID `bulkId`를 필수로 받음.
- 기존 workflow SQLite DB에 벌크 호출과 개별 작업 상태를 영속적으로 기록함.
- 실행 전 모든 작업을 `pending`, 실행 직전을 `running`, 결과를 `succeeded`
  또는 `failed`로 기록함.
- 서버 시작 시 남은 `running`은 결과를 추정하지 않고 `uncertain`으로 변경함.
- `pending`만 명시적으로 재개함.
- uncertain 읽음·별표는 현재 flags를 확인하여 안전하게 확정 또는 재실행함.
- uncertain 복사·이동·휴지통·스팸은 자동 재시도하지 않음.
- 동일 `bulkId` 재호출은 기존 상태만 반환하며 다른 입력에 재사용하면 거부함.
- 위치 변경 계열 성공 결과의 목적지 편지함과 UID를 작업 결과 payload로
  보존함.
- 완료 저널은 30일 동안 보존함.

## 결과

장점:

- 응답 유실과 서버 재시작 후에도 작업별 상태를 조회할 수 있음.
- 시작하지 않은 작업만 안전하게 재개할 수 있음.
- 불확실한 복사 작업의 자동 재시도로 중복 메일을 만들지 않음.

단점:

- 모든 이메일 변경 호출자가 사전에 UUID `bulkId`를 생성하고 보존해야 함.
- 정확히 한 번 실행을 보장하지 않으며 uncertain 작업은 사용자 판단이 필요함.
- workflow SQLite DB의 쓰기와 보존 범위가 늘어남.
