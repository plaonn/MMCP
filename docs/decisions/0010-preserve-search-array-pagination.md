# 검색 배열 응답을 유지하는 UID 페이지네이션

## 상태

승인됨

## 배경

`search_emails`는 기존에 이메일 메타데이터 배열을 직접 반환함. 페이지네이션
정보를 포함하기 위해 응답을 객체로 변경하면 기존 MCP 호출자와 내부
`EmailReader` 계약이 깨짐.

## 결정

- `search_emails`의 기존 `EmailSummary[]` 응답을 유지함.
- 페이지네이션은 편지함별 IMAP UID를 직접 사용하는 exclusive
  `olderThanUid` 입력으로 제공함.
- 호출자는 반환 배열의 마지막 UID를 다음 호출의 `olderThanUid`로 사용함.
- 정확한 `hasMore`와 명시적인 다음 cursor는 반환하지 않음.
- 별표와 inclusive 크기 범위 조건은 UID cursor와 함께 IMAP 서버 검색
  단계에서 처리함.
- 정확한 페이지 상태가 실제로 필요해지면 기존 도구 응답을 변경하지 않고
  별도 도구 추가를 검토함.

## 결과

- 기존 `search_emails` 호출자와 응답 처리가 호환성을 유지함.
- 별도 cursor encoding이나 서버 측 페이지 상태 저장이 필요하지 않음.
- 결과 개수가 요청한 `limit`과 같을 때 다음 페이지 존재 여부를 확인하려면
  추가 호출이 필요할 수 있음.
- UID는 편지함별 식별자이므로 페이지 호출 사이에 편지함과 검색 조건을
  동일하게 유지해야 함.

## 검토한 대안

- 기존 응답을 `{ items, hasMore, nextOlderThanUid }` 객체로 변경: 명시적인
  페이지 정보를 제공하지만 기존 호출자를 깨뜨리므로 제외함.
- 입력 옵션에 따라 배열 또는 객체 반환: output schema와 호출자 처리를
  불필요하게 복잡하게 하므로 제외함.
- offset 페이지네이션: 새 메일 도착과 메시지 이동 시 결과 중복과 누락이
  발생하기 쉬워 제외함.
